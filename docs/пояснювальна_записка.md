# ВСТУП

В сучасних умовах розвитку інформаційних технологій та штучного інтелекту автоматизація процесів аналізу та прогнозування стає ключовим інструментом у різних сферах діяльності, зокрема у спортивній індустрії. Прогнозування результатів спортивних подій є складною задачею, яка потребує обробки великих обсягів даних, врахування множини факторів та застосування сучасних методів машинного навчання.

Актуальність розробки інформаційної системи для прогнозування результатів спортивних подій обумовлена кількома чинниками. По-перше, зростаюча кількість спортивних даних створює потребу в автоматизованих системах їх обробки та аналізу. По-друге, застосування технологій штучного інтелекту дозволяє виявляти приховані закономірності та тренди, недоступні при ручному аналізі. По-третє, така система може бути корисною для спортивних аналітиків, тренерів, журналістів та вболівальників.

Метою курсової роботи є проектування та розробка інформаційної системи «SportPredict AI» — веб-додатку для прогнозування результатів спортивних подій з використанням методів машинного навчання та великих мовних моделей (LLM). Система базується на клієнт-серверній архітектурі з використанням FastAPI (Python) для серверної частини, React.js для клієнтської частини та PostgreSQL як системи управління базами даних.

У процесі виконання курсової роботи вирішуються такі завдання:
- аналіз предметної області та існуючих аналогів;
- моделювання бізнес-процесів за допомогою стандартів IDEF0, DFD, IDEF3, UML;
- концептуальне проектування бази даних;
- розробка серверної частини з REST API;
- реалізація модуля штучного інтелекту для прогнозування;
- розробка інтерактивного веб-інтерфейсу;
- тестування та верифікація системи.

Робота виконана відповідно до методології Agile з використанням ітеративного підходу до розробки, що дозволяє поетапно нарощувати функціональність системи та адаптуватися до змін вимог.

# 1. ПОСТАНОВКА ЗАДАЧІ

## 1.1 Опис предметної області

Предметна область даної курсової роботи охоплює сферу спортивної аналітики та прогнозування результатів спортивних змагань. Основною проблемою в цій галузі є необхідність обробки великих обсягів статистичних даних для формування обґрунтованих прогнозів щодо результатів майбутніх матчів.

На сьогоднішній день аналіз спортивних подій здебільшого виконується вручну експертами-аналітиками, що має ряд суттєвих недоліків:
- суб'єктивність оцінок, залежність від досвіду та емоційного стану аналітика;
- обмежена кількість факторів, які може одночасно врахувати людина;
- значні витрати часу на збір та обробку статистичних даних;
- відсутність системного підходу до збереження та аналізу історичних даних;
- неможливість швидкого масштабування при зростанні кількості подій.

## 1.2 Мета та задачі

Мета курсової роботи — спроектувати та розробити інформаційну систему «SportPredict AI», яка автоматизує процес прогнозування результатів спортивних подій із застосуванням алгоритмів машинного навчання та великих мовних моделей.

Для досягнення мети необхідно вирішити такі задачі:

3. Побудувати моделі бізнес-процесів (IDEF0, DFD, IDEF3, UML).
4. Спроектувати реляційну базу даних для зберігання спортивних даних.
5. Розробити серверну частину (REST API) на базі FastAPI.
6. Реалізувати модуль машинного навчання для числового прогнозування.
7. Інтегрувати великий мовний модель (Llama 3.3 70B через Groq Cloud) для текстового аналізу.
8. Розробити інтерактивний веб-інтерфейс на React.js.
9. Провести тестування та верифікацію системи.

В результаті виконання курсової роботи буде створено повнофункціональну інформаційну систему, що забезпечує:

# 2. ТЕХНІЧНЕ ЗАВДАННЯ

## 2.1 Загальний опис системи

«SportPredict AI» — це веб-орієнтована інформаційна система клієнт-серверної архітектури для прогнозування результатів спортивних подій. Система поєднує традиційні методи машинного навчання для числового прогнозування та великі мовні моделі для генерації текстового аналізу.

Система складається з трьох основних компонентів:
- **Серверна частина (Backend)** — FastAPI (Python 3.12), що надає REST API;
- **Клієнтська частина (Frontend)** — React.js одностсорінковий додаток (SPA);
- **База даних** — PostgreSQL 16 для зберігання всіх даних системи.

## 2.2 Глосарій термінів

| Термін | Визначення |
|--------|-----------|
| REST API | Архітектурний стиль взаємодії компонентів розподіленої системи через HTTP-протокол |
| SPA | Single Page Application — односторінковий веб-додаток |
| ML | Machine Learning — машинне навчання |
| LLM | Large Language Model — великий мовний модель |
| JWT | JSON Web Token — стандарт створення токенів доступу |
| ORM | Object-Relational Mapping — технологія програмування, що зв'язує бази даних з концепціями об'єктно-орієнтованих мов |
| CRUD | Create, Read, Update, Delete — базові операції над даними |
| FastAPI | Сучасний фреймворк для створення API на Python |
| React.js | JavaScript-бібліотека для створення інтерфейсів користувача |
| PostgreSQL | Об'єктно-реляційна система управління базами даних |
| scikit-learn | Бібліотека машинного навчання для Python |
| Groq Cloud | Хмарна платформа для інференсу LLM-моделей |
| IDEF0 | Integration DEFinition — стандарт функціонального моделювання |
| DFD | Data Flow Diagram — діаграма потоків даних |
| UML | Unified Modeling Language — уніфікована мова моделювання |

## 2.3 Опис предметної області

Предметна область — мультиспортивна аналітика з підтримкою шести видів спорту: футбол, баскетбол, теніс, хокей, волейбол та кіберспорт. Система оперує такими предметними сутностями:

- **Спортивні дисципліни** — види спорту, за якими ведеться прогнозування;
- **Команди** — спортивні клуби з характеристиками (назва, місто, рік заснування);
- **Гравці** — спортсмени з прив'язкою до команд (ім'я, позиція, національність);
- **Сезони** — часові періоди проведення чемпіонатів;
- **Матчі** — спортивні зустрічі між командами (дата, місце, рахунок, статус);
- **Статистика команд** — агреговані показники за сезон (перемоги, нічиї, голи тощо);
- **Очні зустрічі** (Head-to-Head) — історія протистоянь між двома командами;
- **AI-моделі** — навчені моделі машинного навчання з метриками якості;
- **Прогнози** — результати прогнозування з ймовірностями та AI-аналізом.

Вимоги до предметної області:
- зберігання повної історії матчів та результатів;
- підтримка множинних сезонів з окремою статистикою;
- забезпечення цілісності даних (команди одного виду спорту, узгодженість рахунків);
- можливість масштабування на інші види спорту.

## 2.4 Перелік учасників системи

Система передбачає три ролі користувачів:

| Роль | Опис | Права доступу |
|------|------|---------------|
| **Адміністратор** (admin) | Управління всією системою | Повний доступ: CRUD команд, матчів, управління AI-моделями, тренування моделей |
| **Аналітик** (analyst) | Аналіз даних та прогнозування | Перегляд даних, створення прогнозів, перегляд статистики |
| **Користувач** (user) | Базовий доступ | Перегляд матчів, команд, статистики, створення прогнозів |

## 2.5 Перелік компонентів системи

**Серверні компоненти:**
1. **FastAPI Application** — основний сервер додатку з REST API;
2. **SQLAlchemy ORM** — асинхронний рівень доступу до бази даних;
3. **PostgreSQL 16** — реляційна СУБД (Docker);
4. **AI Module** — модуль прогнозування: LLM Analyzer + Live Analyzer + Sport Profiles;
5. **LLM Analyzer** — інтеграція з OpenRouter API (первинний) та Google Gemini (резервний);
6. **Live Analyzer** — потокова SSE-аналітика поточних матчів;
7. **Live Poller** — фоновий опитувач ESPN кожні 15 с (оновлення рахунку, подій);
8. **Event Bus** — pub/sub шина для доставки live-оновлень SSE-підписникам;
9. **Auth Service** — JWT-автентифікація + bcrypt + скидання пароля через email;
10. **Background Jobs** — фонові задачі: автозавершення матчів, оновлення статистики;
11. **News Scraper** — збір новин (ESPN News API, Google News RSS, Reddit RSS);
12. **Avatar Scraper** — генерація SVG-аватарів гравців та кешування;
13. **Data Import** — мегаскрапер ESPN (футбол, баскетбол, теніс, хокей, волейбол) + OpenDota (кіберспорт);
14. **Rate Limiter** — обмеження частоти запитів (slowapi);
15. **Docker Compose** — оркестрація контейнерів (3 сервіси: db, backend, frontend).

**Клієнтські компоненти:**
1. **React SPA** — односторінковий веб-додаток з 15 маршрутами;
2. **React Router** — маршрутизація з lazy loading та захищеними маршрутами;
3. **SportModeContext** — глобальний контекст спортивного режиму з 6 темами;
4. **Axios** — HTTP-клієнт з JWT-interceptor та авто-logout при 401;
5. **Recharts** — візуалізація: PieChart для ймовірностей прогнозів;
6. **Tailwind CSS** — фреймворк стилізації з CSS-змінними для динамічних тем;
7. **EventSource (SSE)** — клієнт серверних подій для live-трансляцій матчів;
8. **Toast-сповіщення** — система зворотного зв'язку;
9. **Lucide React** — іконки інтерфейсу.

# 3. ОГЛЯД ЛІТЕРАТУРИ

## 3.1 Вибір методології проектування

Для проектування інформаційної системи «SportPredict AI» обрано методологію **Agile** (гнучка розробка) з елементами **Scrum**. Вибір обумовлений наступними факторами [1]:
- ітеративний підхід дозволяє поетапно нарощувати функціональність;
- гнучкість до змін вимог протягом розробки;
- фокус на робочому продукті, а не на документації;
- можливість швидкого прототипування та тестування.

Модель життєвого циклу — **ітеративна інкрементальна**, що передбачає розробку системи серіями ітерацій, кожна з яких додає нову функціональність [2]. В рамках курсового проєкту виділено наступні ітерації:

1. **Ітерація 1** — проектування бази даних та базового API;
2. **Ітерація 2** — реалізація модуля машинного навчання;
3. **Ітерація 3** — інтеграція LLM та розробка фронтенду;
4. **Ітерація 4** — тестування, оптимізація, документування.

## 3.2 Порівняння методологій

| Критерій | Waterfall | Agile/Scrum | RUP | RAD |
|----------|-----------|-------------|-----|-----|
| Гнучкість до змін | Низька | Висока | Середня | Висока |
| Документування | Повне | Мінімальне | Повне | Мінімальне |
| Ітеративність | Ні | Так | Так | Так |
| Залучення замовника | Низьке | Високе | Середнє | Високе |
| Придатність для невеликих проєктів | Ні | Так | Ні | Так |
| Управління ризиками | Слабке | Сильне | Сильне | Середнє |
| Швидкість отримання результату | Низька | Висока | Середня | Висока |

**Обґрунтування вибору Agile:** курсовий проєкт є відносно невеликим, потребує швидких результатів та можливості адаптації. Agile забезпечує найкращий баланс між швидкістю розробки та якістю продукту для даного класу систем [3].

## 3.4 Аналіз технологій розробки

### Backend — FastAPI (Python)

FastAPI — сучасний, високопродуктивний фреймворк для створення API на Python [7]. Обрано за такими перевагами:
- автоматична документація API (Swagger/OpenAPI);
- асинхронна обробка запитів (async/await);
- вбудована валідація даних через Pydantic;
- типізація на рівні мови;
- екосистема Python для ML (scikit-learn, NumPy, pandas).

### Frontend — React.js

React.js — JavaScript-бібліотека для побудови інтерфейсів користувача [8]. Обрано за:
- компонентний підхід для повторного використання UI-елементів;
- віртуальний DOM для оптимальної продуктивності;
- велика екосистема бібліотек (React Router, Recharts);
- широка підтримка спільноти.

### СУБД — PostgreSQL 16

PostgreSQL — потужна об'єктно-реляційна СУБД з відкритим кодом [9]. Обрано за:
- надійність та відповідність стандартам SQL;
- підтримка складних запитів та транзакцій;
- розширена система типів даних;
- асинхронний драйвер asyncpg для Python.

### ML — scikit-learn

scikit-learn — бібліотека машинного навчання для Python [10]. Використано алгоритми:
- Logistic Regression — лінійна модель для класифікації;
- Random Forest — ансамбль дерев рішень;
- Gradient Boosting — ітеративний ансамблевий метод.

### LLM — OpenRouter + Google Gemini

Для генерації текстового аналізу використовується дворівнева схема провайдерів:
- **OpenRouter API** (первинний) — единий API-шлюз до десятків LLM-моделей. Для аналізу матчів застосовується `nvidia/nemotron-3-super-120b`, для поточного коментаря — `meta-llama/llama-3.3-70b-instruct` (безкоштовні рівні);
- **Google Gemini 2.0 Flash** (резервний) — активується автоматично, якщо `OPENROUTER_API_KEY` не налаштовано.

Обидва провайдери використовують OpenAI-сумісний API (бібліотека `openai` Python), тому перемикання між ними не потребує змін коду.

# 5. КОНЦЕПТУАЛЬНЕ ПРОЕКТУВАННЯ

## 5.1 Модель даних

Для проектування бази даних обрано **реляційну модель даних**, що забезпечує:
- структуровану організацію даних у вигляді таблиць;
- підтримку зв'язків між сутностями через зовнішні ключі;
- забезпечення цілісності даних через обмеження (constraints);
- ефективне виконання складних запитів за допомогою SQL.

СУБД PostgreSQL 16 обрано як цільову платформу за її надійність, відповідність стандартам та підтримку розширених типів даних [9].

## 5.5 Правила цілісності та безпеки

### Правила цілісності:
1. **Каскадне видалення** — видалення команди каскадно видаляє гравців, статистику та зв'язані матчі;
2. **Унікальність** — пара (team_id, season_id) у team_statistics є унікальною;
3. **Обмеження значень** — статус матчу може бути лише з переліку (scheduled, live, completed, cancelled);
4. **Непорожні поля** — username, email, password_hash, match_date є обов'язковими;
5. **Зовнішні ключі** — всі FK-посилання забезпечують референційну цілісність.

### Безпека даних:
1. **Хешування паролів** — використання bcrypt (cost factor = 12) для хешування;
2. **JWT-автентифікація** — токени з обмеженим часом дії (24 години);
3. **Рольова модель доступу** — розмежування прав admin/analyst/user;
4. **CORS-політика** — дозволені лише визначені домени клієнта;
5. **Валідація вхідних даних** — Pydantic-схеми валідують усі вхідні дані на рівні API.

### Приклад коду — витягування ознак

Модуль `feature_extractor.py` формує вектор з 23 числових ознак для ML-моделі:

```python
async def extract_features_for_match(db: AsyncSession, match_id: int) -> dict:
    match = await db.execute(select(Match).where(Match.id == match_id))
    match = match.scalar_one_or_none()
    
    home_stats = await _get_team_stats(db, match.home_team_id, match.season_id)
    away_stats = await _get_team_stats(db, match.away_team_id, match.season_id)
    h2h = await _get_h2h(db, match.home_team_id, match.away_team_id)
    
    return {
        'home_points': home_stats.get('points', 0),
        'away_points': away_stats.get('points', 0),
        'points_diff': home_stats.get('points',0) - away_stats.get('points',0),
        'home_wins': home_stats.get('wins', 0),
        'away_wins': away_stats.get('wins', 0),
        # ... (23 ознаки загалом)
        'home_home_win_rate': home_stats.get('home_win_rate', 0.33),
        'away_away_win_rate': away_stats.get('away_win_rate', 0.33),
    }
```

## 6.3 Модуль штучного інтелекту

### LLM-прогнозування (модуль `llm_analyzer.py`)

Для генерації передматчевого текстового аналізу система використовує OpenRouter API (первинний) та Google Gemini 2.0 Flash (резервний). Процес складається з:

1. **Формування промпту** — структурований запит українською мовою з:
   - статистикою обох команд за сезон (PPG, GD, відсоток перемог);
   - даними очних зустрічей (H2H);
   - стартовими складами (якщо доступні);
   - профілем виду спорту (мітки, правило нічиєї).

2. **Виклик API** — асинхронний виклик через `AsyncOpenAI` з OpenAI-сумісним інтерфейсом. Обидва провайдери використовують однакову бібліотеку `openai`.

3. **Структура відповіді LLM:**
   - Огляд матчу — значимість зустрічі;
   - Аналіз сил — порівняння команд;
   - Ключові фактори — що вплине на результат;
   - Прогноз — експертна думка з обґрунтуванням.

4. **Обробка помилок** — при відсутності API-ключів або недоступності сервісу система повертає `None` — прогноз зберігається без LLM-аналізу.

### Live-аналітика (модуль `live_analyzer.py`)

Для потокового Live-коментаря використовується `meta-llama/llama-3.3-70b-instruct` (через OpenRouter). Модуль генерує текст токен за токеном (`stream=True`) та передає їх клієнту через SSE-потік (`ai_token` events). Коментар описує поточний рахунок, події матчу та оновлені ймовірності.

### Мультиспортивні профілі

Система підтримує специфіку кожного виду спорту через модуль `sport_profiles.py`:

|------------|-------|-----------------|-------------------|----------------|

| 🏀 Баскетбол | Ні | Домашні | Виїзні | Очки |
| 🎾 Теніс | Ні | Гравець 1 | Гравець 2 | Сети |
| 🏒 Хокей | Так | Домашні | Виїзні | Голи |

### Фонові задачі

Система включає автоматичні фонові процеси:

1. **Автозавершення матчів** — симуляція результатів для минулих запланованих матчів на основі ймовірнісної моделі;
2. **Оновлення статистики** — автоматичний перерахунок TeamStatistics та HeadToHead після завершення матчів;
3. **Live-опитувач ESPN** — `live_poller.py` опитує ESPN кожні 15 с, виявляє зміни рахунку/подій та публікує дельта в EventBus;
4. **Оновлення даних ESPN** — періодичний імпорт нових матчів та оновлення існуючих;
5. **Оновлення новин** — `run_news_refresh_loop` щогодини збирає новини для найближчих матчів.

### Імпорт спортивних даних

Реалізовано два модулі імпорту:

1. **ESPN мегаскрапер** (`mega_scraper.py`) — імпорт футболу, NBA, хокею, волейболу, тенісу через ESPN v2 скорборд API. Ростери гравців імпортуються окремо через `roster_scraper.py`.

2. **OpenDota скрапер** (`esports_scraper.py`) — імпорт матчів Dota 2 з відкритого OpenDota API.

## 6.5 Інтерфейс користувача

### Сторінка входу
Градієнтний фон, форма логіну/реєстрації з Toggle-перемикачем, демо-облікові дані для тестування. Анімоване перемикання між режимами.

### Дашборд
8 інформаційних карток зі статистикою (матчі, команди, прогнози, точність). Кільцева діаграма розподілу матчів за статусом. Список найближчих матчів. Таблиця останніх прогнозів з кольоровими індикаторами впевненості.

### Хаб видів спорту
Сітка карток шести видів спорту (⚽ Футбол, 🏀 Баскетбол, 🎾 Теніс, 🏒 Хокей, 🏐 Волейбол, 🎮 Кіберспорт) з навігацією до матчів кожного виду.

### Список матчів
Фільтрація за вкладками (усі/заплановані/завершені). Картки матчів з назвами команд, рахунком, датою та статусом. Пошук за назвами команд. Кольорові бейджі для різних статусів.

### Деталі матчу
Градієнтний хедер з даними про матч. Блок AI-прогнозу з кільцевою діаграмою та прогрес-барами. Розгорнутий текстовий аналіз від LLM. Історія попередніх прогнозів. Toast-сповіщення при помилках.

### Список команд
Сітка карток команд з їх логотипами (перша літера), містами та роками заснування. Пошук за назвою/містом, фільтрація за видом спорту.

### Профіль команди
Градієнтний хедер з назвою та характеристиками. 8 карток статистичних показників. Таблиця статистики по сезонах. Список гравців з позиціями та національностями.

### Турнірна таблиця
Вибір сезону з випадаючого списку. Рейтинг команд з показниками (очки, перемоги, нічиї, поразки, голи, різниця голів). Індикатори форми за останні 5 матчів.

### Історія прогнозів
Розгортувані картки з бейджами результатів, кільцевими індикаторами впевненості. Розгорнутий вигляд із трьома картками ймовірностей та повним AI-аналізом.

### Управління AI-моделями
Картки доступних алгоритмів з кнопками тренування. Тренування найкращої моделі для кожного виду спорту. Список навчених моделей з метриками якості (accuracy, precision, recall, F1). Кнопки активації/деактивації. Діагностика: rolling backtest та ablation study.

### Профіль користувача
Інформація про обліковий запис (ім'я, email, роль). Персональна статистика прогнозів (кількість, точність). Історія прогнозів з індикаторами правильності.

### Сторінка 404
Стильна сторінка з великим текстом «404», повідомленням українською мовою та кнопкою повернення на головну сторінку.

# ВИСНОВКИ

В результаті виконання курсової роботи спроектовано та розроблено інформаційну систему «SportPredict AI» для прогнозування результатів спортивних подій з використанням штучного інтелекту.

У процесі роботи було вирішено наступні задачі:

2. **Виконано моделювання бізнес-процесів** за допомогою стандартів IDEF0 (функціональна модель з декомпозицією до 3-го рівня), DFD (діаграми потоків даних), IDEF3 (діаграми процесів) та UML (діаграми прецедентів, послідовностей та класів).

3. **Спроектовано реляційну базу даних** з 10 таблицями у PostgreSQL 16. Визначено зв'язки між сутностями, правила цілісності та безпеки даних. Побудовано ER-діаграму у нотації IDEF1X.

4. **Розроблено серверну частину** на FastAPI з 7 роутерами та 33 REST API ендпоінтами. Реалізовано асинхронну взаємодію з базою даних через SQLAlchemy та asyncpg. Забезпечено JWT-автентифікацію з рольовою моделлю доступу. Додано rate limiting для захисту від зловживань, health check ендпоінт та structured logging.

5. **Реалізовано модуль штучного інтелекту**, що поєднує:

6. **Розроблено інтерактивний веб-інтерфейс** на React.js з використанням Tailwind CSS для дизайну, Recharts для візуалізації даних та React Router для маршрутизації. Інтерфейс містить 12 сторінок (включаючи хаб видів спорту, турнірну таблицю, профіль користувача та сторінку 404) та забезпечує зручну взаємодію з усіма функціями системи. Реалізовано систему toast-сповіщень для зворотного зв'язку.

7. **Проведено тестування системи**: функціональне, інтеграційне та тестування безпеки. Всі тести пройдено успішно. Система коректно виконує прогнозування, генерує AI-аналіз та відображає результати у веб-інтерфейсі.
